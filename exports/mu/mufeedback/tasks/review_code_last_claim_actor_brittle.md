# REVIEW: lastClaimActor is a brittle prefix-match against free-text event payloads

## Notes (1)

### #1 by "code-reviewer-1", 2026-05-08T11:34:13.941Z

```
FILES:
  src/cli.ts:1812-1834 (lastClaimActor)
  src/tasks.ts:1167-1180, 1252-1259 (the emitEvent call sites that lastClaimActor parses)

FINDINGS: lastClaimActor identifies the last actor for a task by scanning the latest 100 kind=event log rows in the workstream and string-matching `payload.startsWith(`task claim ${localId} `)`. This is a regex-on-prose protocol where:

  - The producer (claimTask in src/tasks.ts) writes free-text:
      `task claim ${localId} by ${agentName} (was owner=...)`
      `task claim ${localId} by ${actor} --self (anonymous, owner stays NULL...)`
  - The consumer (lastClaimActor in cli.ts) prefix-matches on a substring of that string and assumes ev.source carries the actor name.

Failure modes:
  1. localId can collide with a release/reap/reparent payload that ALSO starts with `task ...`. Not today (close/release start with different verbs) but the protocol provides no guarantee. A future "task claim-back" verb whose payload is `task claim-back foo by ...` would silently match `task claim foo `'s prefix on a `localId` that is the prefix of another id (e.g. `foo` as a prefix of `foo_v2`). Wait — `${localId} ` includes the space, so `task claim foo ` won't prefix-match `task claim foo_v2 `. Good. But the pattern is one renaming away from breaking — any payload format change (e.g. switching to "task.claim" or adding a colon) silently returns null actor for every task.

  2. limit=100 is hard-coded. A workstream with ≥100 events since the last claim returns null silently, so `mu task show` says "(unowned)" even though there IS an actor — just not in the recent window. Real failure mode: a long-running workstream where someone --self-claimed a task months ago. The "(self: ...)" attribution silently disappears.

  3. Cross-cutting concern in the wrong layer. The "who claimed this task" question should be answered by the tasks domain (src/tasks.ts), not by parsing event payloads in cli.ts. The data already exists in agent_logs but the schema doesn't index it.

WHY IT MATTERS: silent loss of attribution metadata. The whole point of the --self anonymous-claim feature is that the actor IS recorded — `mu task show` is the verb that surfaces it. When this regex pattern misses, the user sees "(unowned)" and is misled about who's working on what.

SUGGESTED FIX (~30 LOC, several layers):
  Smallest: capture the localId in a structured payload and parse it back. E.g. switch the emitted event payload from prose to a tagged string the consumer can split:

    payload = `task.claim ${localId}	actor=${actor}	self=${anon ? "1" : "0"}`

  Then lastClaimActor splits on 	 and reads the actor field. Still a pattern-match but now it's a delimiter parse, not prose.

  Better: add a `task_claims` table (id, task_id, actor, claimed_at, anonymous) populated by claimTask. lastClaimActor becomes a ROWID-indexed SELECT MAX. The agent_logs entry becomes the audit trail; task_claims becomes the queryable surface. ~50 LOC across schema + tasks.ts; aligns with the schema-versioning machinery already in place.

  In either case, drop the limit=100 ceiling.

ALTERNATIVES CONSIDERED:
  - "raise limit to 10000": kicks the can; doesn't fix the prose-parse fragility.
  - "store the last actor in tasks.last_actor": adds a column, but then "actor" is mutable state that can drift from agent_logs. Audit trail in agent_logs stays the source of truth.

EVIDENCE: src/cli.ts:1830 uses a startsWith on a prose payload generated by src/tasks.ts:1170. No test covers the >100-events-since-claim case (test/tasks.test.ts most likely tests the 1-event case). A renaming of the emitEvent payload string in either path would silently break this.
```
